# 프로젝트 목표

## 한 줄 요약
TeamMaker를 실제로 파일을 읽고 쓰며 빌드 에러를 반복 수정하는 AI 코딩 팀 오케스트레이터로 발전시킨다.

## 목표 상세
### 핵심 목표
1. Electron IPC 의존을 제거하고, 모든 서버사이드 기능을 Next.js API Routes로 이전한다 (Electron은 런처 역할만)
2. 에이전트가 로컬 파일시스템을 직접 읽고 쓰며, 빌드 에러를 감지하고 자동 수정하는 핑퐁 루프를 구현한다
3. 오류 수정이나 추가 요청 시 전체 파이프라인이 아닌 관련 에이전트만 호출하는 부분 실행을 지원한다
4. 초급(문과)/중급(컴공생)/고급(개발자) 모드로 사용자 수준에 맞는 경험을 제공한다

### 성공 기준
- `npm run dev`로 브라우저에서 전체 기능이 동작한다 (FS, 터미널, Claude API 포함)
- `npm run electron:dev`로 Electron 앱에서 동일 기능이 동작한다
- 프로젝트 내 `isElectron()` 호출이 0건이다
- 에이전트가 생성한 코드가 자동으로 프로젝트 디렉토리에 저장되고, 빌드 실패 시 에이전트가 에러를 읽고 수정한다
- 간단한 수정 요청에 전체 팀이 아닌 담당 에이전트 1~2명만 동작한다
- 매니저→팀 내부 메시지가 채팅 UI에 실시간으로 표시된다

### 범위 밖 (Non-goals)
- electron-builder를 이용한 프로덕션 패키징 (DMG/EXE 배포)
- 자동 업데이트 시스템
- 클라우드 배포 / 다중 사용자 지원
- 데이터베이스 도입 (localStorage/파일 기반 유지)

## 배경 및 맥락
현재 TeamMaker는 Electron IPC를 통해 모든 서버사이드 기능을 처리한다. 새 기능 추가 시 IPC 핸들러 + preload 노출 + 타입 정의의 3중 작업이 필요하고, 브라우저에서는 제한된 기능만 사용 가능하다. Next.js API Routes로 통일하면 단일 코드패스로 브라우저/Electron 모두 지원하며, FS와 터미널 접근이 자유로워져 "코드 생성 → 저장 → 빌드 → 에러 수정"까지 자동화할 수 있다.

## 제약 조건
- 기존 UI 코드(React, PixiJS, Zustand)를 최대한 유지
- Electron은 런처로 유지 (사용자가 `npm` 명령을 직접 실행하지 않도록)
- Next.js App Router + API Routes 사용
- 터미널 스트리밍은 SSE(Server-Sent Events)로 구현
