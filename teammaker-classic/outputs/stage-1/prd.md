# TeamMaker PRD (Product Requirements Document)

**버전**: 1.0
**작성일**: 2026-02-10
**상태**: MVP 기획 완료

---

## 1. 제품 개요

### 한 줄 설명
> Gather 스타일 가상 오피스에서 책상을 배치해 AI 에이전트 팀을 만들고, 자연어로 일을 시키면 어떤 팀이 처리 중인지 실시간으로 보여주는 서비스

### 비전
AI를 처음 접하는 비개발자도 "나만의 AI 회사"를 만들고 운영할 수 있는 세상

### 핵심 가치
| 가치 | 설명 |
|------|------|
| **직관성** | 팀을 만드는 익숙한 메타포로 AI를 이해 |
| **가시성** | AI 작업 과정을 실시간으로 시각화 |
| **접근성** | 코딩 없이 자연어만으로 모든 것을 조작 |
| **재미** | 게임 같은 경험으로 진입 장벽을 제거 |

---

## 2. 문제 정의

### 핵심 문제
사람들은 AI 에이전트를 "팀"처럼 조합해서 활용할 수 있다는 **가능성 자체를 인식하지 못하며**, 안다고 해도 설정이 어렵고 과정이 보이지 않아 제대로 활용하지 못한다.

### 문제 구조

| 레벨 | 문제 | 현재 상태 | TeamMaker의 해결 |
|------|------|-----------|-----------------|
| 인식 | AI를 팀으로 쓸 수 있다는 생각을 못 함 | 1:1 채팅만 앎 | "팀 만들기" 메타포 |
| 진입 | 에이전트 설정이 어려움 | 비개발자 불가 | 드래그 + 자연어 |
| 경험 | AI가 뭘 하는지 안 보임 | 블랙박스 | 실시간 시각화 |

### 타겟 사용자
- **페르소나**: 민지 (29세, 스타트업 마케팅 담당)
- AI 도구는 ChatGPT 정도만 써봄
- 코딩 경험 없음, 기술 용어에 거부감
- 반복 업무가 많아서 AI에 위임하고 싶지만 방법을 모름
- "AI가 더 많은 걸 할 수 있다는데 뭘 어떻게 해야 할지 모르겠다"

## 4. 사용자 스토리

### Epic 1: 가상 오피스

**US-1.1: 오피스 둘러보기**
> 사용자로서, 서비스에 접속하면 빈 사무실 공간이 보이고, 여기에 내 AI 팀을 채울 수 있다는 걸 직관적으로 알 수 있어야 한다.

- 접속 시 2D 탑뷰 오피스 공간 표시
- 마우스 드래그로 패닝, 스크롤로 줌
- "여기에 팀을 배치해보세요" 안내 메시지

### Epic 2: 팀 생성

**US-2.1: 책상 배치하기**
> 사용자로서, 화면 한쪽에 있는 빈 책상을 드래그해서 오피스 원하는 위치에 놓을 수 있어야 한다.

- 팔레트에서 빈 책상 드래그 → 오피스에 드롭
- 그리드 스냅으로 깔끔한 배치
- 드롭 즉시 팀 정보 입력 폼 오픈

**US-2.2: 팀 설명하기**
> 사용자로서, "이 팀이 뭘 하는 팀인지"를 내 말로 자유롭게 설명하면 AI가 알아서 이해해야 한다.

- 팀 이름 + 설명 입력
- 예시 플레이스홀더 제공
- "팀 만들기" 버튼으로 AI 분석 시작

**US-2.3: AI 팀 구성 확인하기**
> 사용자로서, AI가 제안한 에이전트 구성을 보고 이해할 수 있어야 한다.

- AI가 에이전트 구성 자동 제안
- 각 에이전트 역할이 쉬운 말로 설명됨
- "팀을 구성하고 있어요..." 로딩 피드백

**US-2.4: 팀 구성 수정하기**
> 사용자로서, AI가 제안한 팀 구성이 맘에 안 들면 수정할 수 있어야 한다.

- 에이전트 삭제/수정/추가 가능
- 수정 후 확인하면 팀 최종 생성

### Epic 3: 업무 지시

**US-3.1: 일 시키기**
> 사용자로서, 채팅창에 하고 싶은 말을 쓰면 AI 팀이 알아서 처리를 시작해야 한다.

- 자연어로 업무 입력
- AI가 적절한 팀 자동 매칭
- 매칭 결과 채팅에 표시
- `@팀이름`으로 직접 지정 가능

**US-3.2: 진행 상황 눈으로 보기**
> 사용자로서, 일을 시키면 오피스에서 어떤 팀이 일하고 있는지 바로 눈에 보여야 한다.

- 작업 중 데스크 시각적 활성화 (글로우)
- 대기/작업중/완료 상태 명확히 구분
- 완료 시 상태 변경

**US-3.3: 상세 진행 상황 확인**
> 사용자로서, 일하고 있는 팀의 데스크를 클릭하면 구체적으로 뭘 하고 있는지 볼 수 있어야 한다.

- 데스크 클릭 → 사이드 패널
- 각 에이전트별 진행 상태 표시
- 완료된 결과물 확인

### Epic 4: 팀 협업 시각화

**US-4.1: 팀 간 일 넘기기 보기**
> 사용자로서, 한 팀이 다른 팀에게 결과를 넘기는 과정이 시각적으로 보여야 한다.

- 팀A → 팀B 결과 전달 시 이동 애니메이션
- 이동 중 라벨로 내용 표시
- 받는 팀 자동 활성화

**US-4.2: 복잡한 업무 위임하기**
> 사용자로서, 큰 업무를 시키면 여러 팀이 순서대로/동시에 협업하는 걸 볼 수 있어야 한다.

- AI가 하위 작업으로 분해
- 팀 간 의존 관계에 따라 순차/병렬 처리
- 전체 진행률 파악 가능

### Epic 5: 기본 설정

**US-5.1: API Key 입력**
> 사용자로서, Anthropic API Key를 입력하면 바로 사용할 수 있어야 한다.

- 최초 접속 시 API Key 입력 화면
- Key 검증 후 오피스로 이동
- 로컬 저장으로 재입력 불필요

### Epic 6: 온보딩

**US-6.1: 첫 팀 만들기 가이드**
> 사용자로서, 처음 빈 사무실을 봤을 때 뭘 해야 하는지 바로 알 수 있어야 한다.

- 팔레트 하이라이트로 시선 유도
- 첫 팀 생성 완료 시 축하 피드백
- 다음 단계 안내

## 5. 사용자 흐름

### Flow 1: 첫 접속 → 첫 팀 생성

```
접속 → API Key 입력 → 빈 오피스 진입
  → 온보딩 안내 표시
  → 팔레트에서 책상 드래그
  → 오피스에 드롭 (그리드 스냅)
  → 팀 정보 입력 패널 오픈
  → 이름/설명 입력 → "팀 만들기"
  → AI 로딩 (2-5초)
  → 에이전트 구성 제안 표시
  → 확인 (또는 수정 후 확인)
  → 팀 생성 완료! (축하 이펙트)
  → "채팅에서 일을 시켜보세요" 안내
```

### Flow 2: 업무 지시 → 결과 확인

```
채팅에 업무 입력 → AI 팀 자동 매칭
  → 매칭 결과 채팅에 표시
  → 해당 팀 데스크 활성화 (글로우)

  [단일 팀 작업]
  → 에이전트들 순차 처리
  → 완료 → 결과물 채팅에 표시

  [복수 팀 협업]
  → 첫 팀 활성화 → 작업 완료
  → 결과물이 다음 팀으로 이동 (애니메이션)
  → 다음 팀 활성화 → 작업
  → ... 반복
  → 최종 완료 → 결과물 표시
```

### 핵심 원칙
- **한 화면 원칙**: 오피스 캔버스가 항상 메인. 패널과 채팅은 오버레이/사이드로 열림
- **페이지 이동 최소화**: 설정 페이지만 별도. 나머지는 모두 오피스 화면에서 처리

## 7. 기술 아키텍처

### 기술 스택

| 레이어 | 기술 | 근거 |
|--------|------|------|
| 캔버스 렌더링 | **PixiJS v8** | Gather.town이 사용하는 동일 기술 |
| React 연동 | **@pixi/react v8** | JSX로 선언적 캔버스 작성 |
| 애니메이션 | **GSAP** | Gather.town과 동일 |
| 실시간 통신 | **Socket.io** | 재연결, 룸 기능 지원 |
| 프레임워크 | **React + TypeScript** | @pixi/react와 호환 |
| 상태 관리 | **Zustand** | 경량, 실시간 상태에 적합 |
| AI | **Anthropic Claude API** | 사용자 API Key 방식 |

### 아키텍처

```
┌──────────── 프론트엔드 (React) ─────────────┐
│  PixiJS 캔버스    ←→    UI 컴포넌트          │
│      │                      │               │
│      └───── Zustand Store ──┘               │
└────────────────┬────────────────────────────┘
                 │ WebSocket + REST
┌────────────────┴────────────────────────────┐
│           백엔드 (Node.js)                    │
│  WebSocket 서버   ←→   에이전트 오케스트레이터  │
│                          │                   │
│                  Claude API (Tool Use)        │
└──────────────────────────────────────────────┘
```

### 기술 타당성: 8/10
- PixiJS: 60fps에서 1000+ 스프라이트 처리 가능
- 오픈소스 레퍼런스: [gather-clone](https://github.com/trevorwrightdev/gather-clone)

## 8. 스프린트 계획

### Sprint 1: 오피스 + 팀 만들기 (2-3주)
> "빈 사무실에 책상을 놓고 AI 팀을 만들 수 있다"

| 기능 | 상세 |
|------|------|
| A1 | 2D 그리드 오피스 캔버스 |
| A2 | 줌/패닝 |
| B1 | 책상 드래그앤드롭 |
| B2 | 팀 이름/설명 입력 |
| B3 | AI 에이전트 구성 제안 |
| B4 | 구성 확인/수정 |
| D1 | API Key 입력/저장 |

**데모**: 빈 오피스 → 책상 배치 → "마케팅팀" 설명 입력 → AI 제안 → 확인 → 오피스에 팀 표시

### Sprint 2: 업무 지시 + 시각화 (2-3주)
> "팀에게 일을 시키고 진행 과정을 볼 수 있다"

| 기능 | 상세 |
|------|------|
| C1 | 채팅 인터페이스 |
| C2 | AI 팀 자동 매칭 |
| C3 | 팀 활성 애니메이션 |
| A3 | 데스크 상태 비주얼 |
| C5 | 상세 진행 패널 |

**데모**: 채팅 입력 → 팀 매칭 → 데스크 빛남 → 클릭 → 진행 상황 → 완료 → 결과

### Sprint 3: 팀 협업 + 폴리싱 (2-3주)
> "여러 팀이 협업하는 모습을 볼 수 있다"

| 기능 | 상세 |
|------|------|
| C4 | 팀 간 협업 시각화 |
| D2 | 온보딩 가이드 |
| D3 | 작업 히스토리 |
| - | 전체 UX 폴리싱 |

**데모**: 복잡한 업무 → 기획팀 활성 → 디자인팀으로 전달(애니메이션) → 마케팅팀 → 최종 결과

## 9. 성공 지표

### 정량 지표
| 지표 | 목표 | 의미 |
|------|------|------|
| 첫 팀 생성 완료율 | > 80% | 온보딩이 직관적인지 |
| 첫 업무 지시 완료율 | > 70% | 팀 생성 후 실제 사용하는지 |
| 세션당 평균 업무 지시 수 | > 3회 | 지속적으로 사용하는지 |
| 평균 팀 수 | > 2개 | 여러 팀을 만드는지 |

### 정성 지표
- "AI가 뭘 하는지 보여서 신뢰가 간다"
- "팀 만드는 게 게임 같아서 재밌다"
- "코딩 몰라도 AI를 쓸 수 있다"
